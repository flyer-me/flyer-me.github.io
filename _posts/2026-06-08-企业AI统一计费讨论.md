# 企业 AI 统一计费讨论

企业使用 AI 后最先暴露的问题通常是管理方式：

常见问题：

- 员工各自注册 AI 账号，费用分散，报销口径不统一
- 具体项目、部门、员工的使用情况难以自动统计
- 员工需要反复切换工具，工具效率降低
- 模型、供应商迁移成本高

基于以上，企业应该解决有一套 AI 统一计费方案。本文重点给出一个参考构想，说明每一层该选什么、为什么选、怎么落地。

## 一、整体结论

建议采用“四层结构”：

```text
AI 采购
  ↓
AI 审计网关
  ↓
统一入口
  ↓
员工使用与项目归集
```

推荐起步组合：

| 层次          | 主选方案                                   | 备选方案                         |
| ------------- | ------------------------------------------ | -------------------------------- |
| AI 采购       | OpenAI、Gemini、Claude、通义千问、DeepSeek | OpenRouter、硅基流动、阿里云百炼 |
| AI 网关与计费 | New API、LiteLLM                           | Higress、云厂商 AI 网关          |
| 统一入口      | Open WebUI                                 | LobeChat、Dify、NextChat         |
| 成本归集      | 项目和个人 Key                             | 部门 Key、员工 Key、日志导出     |
| 专业软件      | 企业版订阅                                 | 支持 API Key 的插件接入 New API  |

一句话方案：

> 用 New API 统一管理模型和费用，用 Open WebUI 作为员工入口，用 Key 把成本归类。

## 二、全链路总览

```text
第一层：AI 采购
OpenAI / Claude / Gemini / 通义千问 / DeepSeek / OpenRouter / 硅基流动
        ↓
第二层：AI 网关与计费
New API / LiteLLM / Higress / 云厂商 AI 网关
        ↓
第三层：统一入口
Open WebUI / LobeChat / Dify / NextChat
        ↓
第四层：员工使用
文案、图片理解、资料分析、方案撰写、客服问答、项目资料整理
        ↓
第五层：财务核算
按项目、部门、员工导出月度报表
```

这套结构的核心是第二层。只要所有 AI 使用都经过统一网关，公司就能知道谁用了、用在哪个项目、花了多少钱。

## 三、第一层：AI 采购

AI 采购层解决“从哪里买模型能力”的问题。

| 方案                         | 优点                               | 缺点                       | 适用场景                       |
| ---------------------------- | ---------------------------------- | -------------------------- | ------------------------------ |
| 直购 OpenAI、Claude、Gemini  | 模型能力强，更新快                 | 付款、网络、发票可能较麻烦 | 需要高质量模型效果             |
| 阿里云百炼、腾讯云、火山方舟 | 国内付款和发票方便，合规性较好     | 国际前沿模型覆盖有限       | 重视合规和对公流程             |
| OpenRouter                   | 一个账号接入多家模型，适合横向比较 | 海外链路和支付可能不稳定   | 试用多模型、快速评估           |
| 硅基流动等国内聚合平台       | 国产模型接入方便，成本较低         | 国际模型覆盖有限           | 主要使用国产模型               |
| 单独购买软件自带 AI          | 与原软件工作流结合好               | 难纳入统一网关计费         | Adobe、Figma、Canva 等专业软件 |

采购建议：

- 不建议员工自行购买后报销
- 公司集中管理模型账号和付款方式
- 起步选择 2 到 3 个主力供应商即可
- 免费或低价模型可以补充，但不要作为唯一依赖
- 供应商必须能提供用量记录或账单明细

推荐起步：

| 公司情况           | 建议采购方式                   |
| ------------------ | ------------------------------ |
| 需要国际模型能力   | OpenAI 或 Gemini 作为主力      |
| 重视国内合规和发票 | 阿里云百炼、腾讯云、火山方舟   |
| 希望快速比较多模型 | OpenRouter 或国内聚合平台      |
| 预算敏感           | 国内模型为主，国际模型按需启用 |

## 四、第二层：AI 网关与计费

AI 网关是整个方案的核心。它负责统一接入模型、分配 Key、记录用量、控制额度和导出报表。

| 方案           | 优点                                        | 缺点                           | 适用场景                     | 起步建议         |
| -------------- | ------------------------------------------- | ------------------------------ | ---------------------------- | ---------------- |
| New API        | 部署轻量，界面清楚，适合统一 Key 和用量统计 | 更复杂的预算和财务集成需要开发 | 非技术公司、小中型团队       | 最小起步         |
| LiteLLM        | 多模型支持强，预算和标签能力更细            | 管理和配置门槛略高             | 对项目预算控制要求更高的团队 | 起步的升级方案   |
| Higress        | 适合云原生和大规模流量治理                  | 计费报表通常需要额外开发       | 已有技术团队和云基础设施     | 非技术公司不建议 |
| 云厂商 AI 网关 | 合规、稳定、可与云账号体系结合              | 绑定云生态，灵活性受限制       | 已深度使用某家云厂商         | 大型企业路线     |
| 自研网关       | 完全可自定义                                | 周期长、成本高、维护压力大     | 大型企业的强技术团队         | 不建议作为起步   |

本层建议：

- 起步选择 New API
- 产生了预算熔断、标签计费、复杂策略的要求，可升级 LiteLLM
- 大型企业和强技术团队使用Higress和云/自研网关

### New API 的推荐用法

在 New API 中建立四类 Key：

| Key 类型 | 用途                         | 计费归属     |
| -------- | ---------------------------- | ------------ |
| 项目 Key | 客户项目、内部项目           | 项目成本     |
| 部门 Key | 市场、运营、设计、行政等部门 | 部门预算     |
| 员工 Key | 高频个人用户                 | 员工使用记录 |
| 测试 Key | 临时验证、新模型测试         | 测试费用     |

起步阶段优先使用“项目 Key”。它最容易解释，也最方便财务核算。

示例：

| 项目           | Key 命名        | 月度额度 | 负责人     |
| -------------- | --------------- | -------- | ---------- |
| A 客户营销项目 | PRJ-A-Marketing | 3000 元  | 项目经理   |
| B 客户设计项目 | PRJ-B-Design    | 2000 元  | 设计负责人 |
| 内部运营       | DEPT-Operation  | 1000 元  | 运营负责人 |
| 临时测试       | TEST-General    | 300 元   | IT 或行政  |

## 五、第三层：统一入口

统一入口解决“员工从哪里使用 AI”的问题。

员工不应该直接拿模型官方 Key，也不应该自己研究供应商配置。员工只需要打开一个公司入口使用 AI。

| 方案           | 优点                                               | 缺点                   | 适用场景                 | 建议                   |
| -------------- | -------------------------------------------------- | ---------------------- | ------------------------ | ---------------------- |
| Open WebUI     | 员工上手快，能接入 New API，适合日常对话和文件分析 | 界面相对朴素           | 通用 AI 办公入口         | 推荐起步               |
| LobeChat       | 界面更现代，体验更好                               | 多用户和权限配置更复杂 | 重视界面体验的团队       | 可作为替代             |
| Dify           | 适合固定流程和业务应用                             | 作为通用聊天入口偏重   | 客服、知识库、自动化流程 | 第二阶段引入           |
| NextChat       | 极简，部署快                                       | 多用户管理弱           | 小团队临时验证           | 不建议作为企业长期入口 |
| 企业 IM 机器人 | 员工在飞书、企微、钉钉内使用                       | 能力和体验受 IM 限制   | 高频轻量问答             | 可作为补充入口         |

本层建议：

- 日常通用入口选 Open WebUI
- 固定业务流程后续用 Dify
- 不建议把 Dify 当作唯一入口
- 不建议让员工直接使用官方 App 作为公司主入口

推荐分工：

| 工具       | 在方案中的位置                    |
| ---------- | --------------------------------- |
| New API    | 管账、管 Key、管模型              |
| Open WebUI | 员工日常使用入口                  |
| Dify       | 固定流程和专项应用                |
| LobeChat   | 对界面体验要求高时替代 Open WebUI |

## 六、第四层：员工使用与项目归集

员工使用层解决“费用归到哪里”的问题。公司最终关心的不是某个模型用了多少，而是哪个项目、部门或员工产生了成本。

| 方案                | 做法                               | 优点         | 缺点                                 | 适用场景                             | 建议             |
| ------------------- | ---------------------------------- | ------------ | ------------------------------------ | ------------------------------------ | ---------------- |
| 项目 Key            | 每个项目分发 Key                   | 项目成本清楚 | 项目多时需要维护 Key，员工需选对项目 | 客户项目、内部专项、可明确归属的工作 | 推荐起步         |
| 部门 Key            | 每个部门使用独立 Key，按部门汇总   | 管理简单     | 无法直接看到项目级成本               | 行政、运营、市场等通用部门           | 项目不清晰时使用 |
| 员工 Key            | 给员工分配独立 Key                 | 可追踪到个人 | 不能天然反映项目成本，管理量较大     | 核心岗位、高频用户、试点负责人       | 作为补充         |
| 项目 Key + 员工 Key | 项目维度看成本，员工维度看使用习惯 | 管理最清楚   | 配置和报表更复杂                     | 使用规模扩大后                       | 第二阶段再做     |

本层推荐：先用项目 Key；如果项目边界不清晰，就先用部门 Key；等使用量稳定后，再补员工 Key。

## 七、第五层：专业软件集成

设计、视频、办公协作类软件通常已经有自己的 AI 功能，不建议强行全部接入统一网关。

| 方案 | 例子 | 优点 | 缺点 | 适用场景 | 建议 |
| --- | --- | --- | --- | --- | --- |
| 企业版统一订阅 | Adobe、Figma、Canva | 成员、额度、发票统一管理 | 费用通常不能进入 New API 报表 | 设计、品牌、视频等专业岗位 | 推荐 |
| 项目账号统一管理 | 剪映、Runway、Vidu 等 | 项目费用较清楚，便于交接 | 账号管理需要专人负责 | 视频、创意、投放项目 | 可用 |
| 插件填入公司 Key | 支持 API Key 的设计、剪辑、自动化插件 | 可纳入 New API 统一计费 | 依赖插件是否支持自定义 Key | 插件能配置接口时 | 推荐接入 |
| 企业 IM 入口 | 飞书、钉钉、企业微信 | 员工触达方便 | 复杂任务体验不如网页入口 | 轻量问答、通知、审批 | 作为补充 |

本层推荐：专业软件优先走企业版；能填公司 Key 的插件接入 New API；无法统一计费的软件，至少统一采购和报销口径。

## 八、推荐组合方案

| 组合 | 适用场景 | AI 采购 | 网关 | 入口 | 计费方式 | 优点 | 缺点 | 建议 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 轻量起步 | 10 人以内，先试用 | OpenRouter 或国内聚合平台 | New API | Open WebUI 或 NextChat | 项目 Key | 上线快，成本低 | 预算和审批能力有限 | 小范围验证 |
| 标准企业 | 10 到 100 人，正式纳入管理 | OpenAI、Gemini、Claude、通义千问、DeepSeek 中选 2 到 3 个 | New API | Open WebUI | 项目 Key + 部门 Key | 员工体验统一，成本可查，不需要自研 | 需要维护 Key 和月度报表 | 推荐主方案 |
| 流程增强 | 已有稳定使用场景 | 主力模型 + 国内模型 | New API | Open WebUI + Dify | 项目 Key + 应用维度 | 可沉淀固定流程 | 应用维护成本更高 | 第二阶段采用 |
| 精细治理 | 100 人以上或有 IT 团队 | 云厂商企业服务 + 国际模型 + 国内模型 | LiteLLM 或 Higress | Open WebUI + Dify | 项目、部门、员工、应用多维度 | 管理精细，可接入公司流程 | 初始复杂度高 | 规模化后再做 |

本层推荐：大多数非技术公司采用“标准企业”组合；不要从自研平台或复杂治理方案起步。

## 九、实施路径

| 阶段 | 目标 | 关键动作 | 验收标准 | 不建议做 |
| --- | --- | --- | --- | --- |
| 第一阶段：跑通闭环 | 先统一入口和计费 | 准备香港或新加坡云服务器；部署 New API；接入 2 到 3 个模型供应商；部署 Open WebUI；创建项目 Key 和部门 Key；选 1 到 2 个部门试用 | 员工能通过统一网页使用 AI；所有调用经过 New API；能看到项目或部门用量；能导出月度报表 | 一开始接入太多模型；一开始做复杂审批 |
| 第二阶段：建立规则 | 让 AI 使用进入公司管理流程 | 明确可上传和不可上传内容；建立项目 Key 申请和停用流程；设置项目和部门月度额度；规定费用归属负责人；建立月度报表模板 | 新项目能快速申请额度；项目结束能停用 Key；财务能按项目或部门看费用；负责人能发现异常用量 | 只看总账单；项目结束后继续保留 Key |
| 第三阶段：扩展自动化 | 把高频工作变成固定流程 | 引入 Dify；选择客服、运营、市场、行政等高频场景；建立专项 AI 应用；继续通过统一网关计费；按应用评估投入产出 | 至少 2 到 3 个固定流程被 AI 承接；每个流程有负责人；费用仍能归属到项目或部门 | 为低频场景做复杂应用；绕过统一网关计费 |

## 十、月度报表建议

建议每月固定输出以下报表。

| 报表 | 用途 | 主要字段 | 输出频率 | 负责人 |
| --- | --- | --- | --- | --- |
| 项目用量报表 | 给老板和项目负责人看 | 项目、模型、调用量、费用、预算、是否超额 | 每月 | 项目负责人或财务 |
| 部门用量报表 | 给部门负责人和财务看 | 部门、员工、调用量、费用、趋势 | 每月 | 部门负责人或财务 |
| 异常用量报表 | 排查误用和超额 | Key、时间、模型、费用、异常原因 | 每周或每月 | 管理员 |
| 供应商账单表 | 对账和付款 | 供应商、模型、账单金额、付款状态、发票状态 | 每月 | 财务 |

本层推荐：报表不需要一开始完全自动化，先保证每月能稳定导出、能看懂、能用于管理决策。

## 十一、选型速查

| 公司情况 | 推荐采购 | 推荐网关 | 推荐入口 | 计费方式 | 建议 |
| --- | --- | --- | --- | --- | --- |
| 10 人以内快速验证 | OpenRouter 或国内聚合平台 | New API | Open WebUI 或 NextChat | 项目 Key | 快速跑通 |
| 10 到 100 人正式使用 | 2 到 3 个主力供应商 | New API | Open WebUI | 项目 Key + 部门 Key | 推荐主方案 |
| 需要固定业务流程 | 主力供应商 + 国内模型 | New API | Open WebUI + Dify | 项目 Key + 应用维度 | 第二阶段采用 |
| 要更细预算控制 | 多模型组合 | LiteLLM | Open WebUI | 标签、项目、部门 | 管理能力升级 |
| 已有技术团队和 K8s | 云厂商企业服务 + 多模型 | Higress 或 LiteLLM | Open WebUI / Dify | 日志与账单系统 | 大规模后采用 |
| 不想自运维 | 云厂商或 SaaS 平台 | 厂商网关 | 厂商入口 | 厂商账单 | 用钱换省事 |

## 十二、最终建议

| 优先级 | 建议 | 原因 |
| --- | --- | --- |
| 必须先做 | New API + Open WebUI + 项目 Key | 先把入口、模型和费用统一起来 |
| 起步不要做 | 自研平台、复杂审批、一次性接入过多模型 | 成本高，周期长，容易拖慢落地 |
| 稳定后再做 | Dify 固定流程、员工 Key、预算提醒 | 等真实使用场景明确后再投入 |
| 规模化后再做 | LiteLLM、Higress、SSO、财务系统对接 | 适合更高使用量和更强治理要求 |

最终建议：先用 New API 把模型、Key 和用量统一起来，用 Open WebUI 给员工提供统一网页入口，用项目 Key 或部门 Key 完成成本归集；使用稳定后，再引入 Dify 做固定业务流程。

这套方案的关键不是工具本身，而是让企业 AI 使用从“个人随便用”变成“公司可管理、项目可核算、费用可控制”。
