# 企业 AI 统一计费讨论

企业使用 AI 后最先暴露的问题通常是管理方式：

常见问题：

- 员工各自注册 AI 账号，费用分散，报销口径不统一
- 具体项目、部门、员工的使用情况难以自动统计
- 员工需要反复切换工具，工具效率降低
- 模型、供应商迁移成本高

基于以上，企业应该解决有一套 AI 统一计费方案。本文重点给出一个参考构想，说明每一层该选什么、为什么选、怎么落地。

## 一、整体结论

建议采用“四层结构”：

```text
AI 采购
  ↓
网关审计
  ↓
统一入口
  ↓
财务核算
```

## 二、全链路总览

```text
第一层：AI 采购
OpenAI / Claude / Gemini / 通义千问 / DeepSeek / OpenRouter / 硅基流动
        ↓
第二层：网关审计
New API / LiteLLM / Higress / 云厂商 AI 网关
        ↓
第三层：统一入口
Open WebUI / LobeChat / Dify / NextChat
        ↓
第四层：财务核算
按项目、部门、员工导出月度报表
```

这套结构的核心是第二层。只要所有 AI 使用都经过统一网关，公司就能知道谁用了、用在哪个项目、花了多少钱。

## 三、第一层：AI 采购

AI 采购层解决“从哪里买模型能力”的问题。

| 方案                                   | 优点                               | 缺点                     | 适用场景                       |
| -------------------------------------- | ---------------------------------- | ------------------------ | ------------------------------ |
| 直购 OpenAI、Claude、Gemini            | 模型能力强，更新快                 | 涉及海外流程较麻烦       | 需要最高质量模型               |
| 阿里云百炼、腾讯云、火山方舟、硅基流动 | 国内付款和发票方便                 | 国际前沿模型覆盖有限     | 重视合规和对公流程             |
| OpenRouter                             | 一个账号接入多家模型，适合横向比较 | 海外链路和支付可能不稳定 | 试用多模型、快速评估           |
| 单独购买软件自带 AI                    | 与原软件工作流结合好               | 难纳入统一网关计费       | Adobe、Figma、Canva 等专业软件 |


## 四、第二层：网关审计

AI 网关是整个方案的核心。它负责统一接入模型、分配 Key、记录用量、控制额度和导出报表。

| 方案           | 优点                                        | 缺点                       | 适用场景                 | 起步建议         |
| -------------- | ------------------------------------------- | -------------------------- | ------------------------ | ---------------- |
| New API        | 部署轻量，界面清楚，适合统一 Key 和用量统计 | 复杂逻辑需要开发           | 非技术公司、小型团队     | 最小起步         |
| LiteLLM        | 预算和标签能力更细                          | 管理和配置门槛略高         | 对预算控制要求更高的团队 | 起步的升级方案   |
| Higress        | 适合云原生和大规模流量治理                  | 计费报表通常需要额外开发   | 已有技术团队和云基础设施 | 非技术公司不建议 |
| 云厂商 AI 网关 | 合规、稳定、可与云账号体系结合              | 绑定云生态，灵活性受限制   | 已深度使用某家云厂商     | 大型企业路线     |
| 自研网关       | 完全可自定义                                | 周期长、成本高、维护压力大 | 大型企业的强技术团队     | 不建议作为起步   |

本层建议：

- 起步选择 New API
- 产生了预算熔断、标签计费（如基于项目）、复杂策略的要求，可升级 LiteLLM
- 大型企业和强技术团队使用Higress和云/自研网关

### 分配 Key 的推荐用法

有四类 Key 可建立：

| Key 类型 | 用途                       | 计费归属     | 示例命名                |
| -------- | -------------------------- | ------------ | ----------------------- |
| 项目 Key | 客户项目、内部项目         | 项目成本     | sk-Project-MB20260101   |
| 部门 Key | 市场、运营、设计、行政部门 | 部门预算     | sk-Department-Operation |
| 员工 Key | 高频个人用户               | 员工使用记录 | sk-Person-202012001     |
| 测试 Key | 临时验证、新模型测试       | 测试费用     | sk-TEST-devteam-202602  |


## 五、第三层：统一入口

统一入口解决“员工从哪里使用 AI”的问题。

员工不应该直接拿模型官方 Key，也不应该自己研究供应商配置。员工只需要打开一个统一入口使用 AI。

| 方案           | 优点                         | 缺点                   | 适用场景                 | 建议                   |
| -------------- | ---------------------------- | ---------------------- | ------------------------ | ---------------------- |
| Open WebUI     | 员工上手快，适合日常         | 界面相对朴素           | 通用 AI 办公入口         | 推荐起步               |
| LobeChat       | 界面更现代，体验更好         | 多用户和权限配置更复杂 | 重视界面体验的团队       | 可作为替代             |
| Dify           | 适合固定流程和业务应用       | 作为通用聊天入口偏重   | 客服、知识库、自动化流程 | 上升阶段引入           |
| NextChat       | 极简，部署快                 | 多用户管理弱           | 小团队临时验证           | 不建议作为企业长期入口 |
| 企业 IM 机器人 | 员工在飞书、企微、钉钉内使用 | 能力和体验受 IM 限制   | 高频轻量问答             | 可作为补充入口         |

本层建议：

- 日常通用入口选 Open WebUI
- 固定业务流程用 Dify
- 其余作为补充


## 六、第四层：财务核算

员工使用层解决“费用归到哪里”的问题。公司最终关心的是哪个项目、部门或员工产生的成本。

本层建议：根据 Key 划分成本；使用企业席位订阅时，根据 席位 划分成本。

## 七、软件的 AI 集成

设计、视频、办公协作类软件通常已经有自己的 AI 功能，不建议接入统一网关。

| 方案             | 例子                                | 优点                     | 缺点                       | 适用场景                   | 建议     |
| ---------------- | ----------------------------------- | ------------------------ | -------------------------- | -------------------------- | -------- |
| 企业版统一订阅   | Adobe、Figma、剪映、Runway、Vidu 等 | 成员、额度、发票统一管理 | 账号管理需要专人负责       | 设计、品牌、视频等专业岗位 | 推荐     |
| 插件填入公司 Key | 支持 API Key 的软件                 | 可纳入网关统一计费       | 依赖插件是否支持自定义 Key | 插件能配置接口时           | 推荐接入 |

本层建议：专业软件优先走企业版；能填 Key 的插件接入网关。实际来说，软件公司的简单做法是开发工具每人提供订阅的月额度。

## 八、推荐组合方案

| 组合     | 适用场景               | AI 采购         | 网关               | 入口              | 计费方式            | 优点                     | 缺点             | 建议         |
| -------- | ---------------------- | --------------- | ------------------ | ----------------- | ------------------- | ------------------------ | ---------------- | ------------ |
| 标准起步 | 10-100 人纳入管理      | 主流模型 2-3 个 | New API            | Open WebUI        | 项目 Key + 部门 Key | 体验统一，不需自研       | 需要维护月度报表 | 推荐主方案   |
| 流程增强 | 已有稳定使用场景       | 模型池子        | New API/LiteLLM    | Open WebUI + Dify | 项目 Key + 应用维度 | 可沉淀固定流程           | 应用维护成本更高 | 第二阶段采用 |
| 精细治理 | 100 人以上，有 IT 团队 | 云厂商企业服务  | LiteLLM 或 Higress | Open WebUI + Dify | 多维度              | 管理精细，可接入公司流程 | 初始复杂度高     | 规模化后再做 |

本层建议：大多数非技术公司采用“标准起步”组合；不要从自研平台或复杂治理方案起步。

## 九、实施路径

| 阶段                 | 目标                       | 关键动作                                                                              | 验收标准                                                                  | 不建议做                               |
| -------------------- | -------------------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | -------------------------------------- |
| 第一阶段：跑通闭环   | 先统一入口和计费           | 准备香港云服务器；部署 New API；接入模型供应商；部署 Open WebUI；创建 Key，选部门试用 | 员工能通过统一网页使用 AI；所有调用经过 网关；能看到 Key 用量，能导出报表 | 一开始接入太多模型；做复杂审批         |
| 第二阶段：建立规则   | 让 AI 使用进入公司管理流程 | 明确内容限制；建立 Key 审批流程；设置额度，归属负责人；建立报表模板                   | 项目快速申请和停用 Key；能多维度看费用；负责人能发现异常用量              | 只看总账单；项目结束后继续保留 Key     |
| 第三阶段：扩展自动化 | 把高频工作变成固定流程     | 引入 Dify；选择客服、运营等高频场景；建立专项 AI 应用；按应用评估投入产出             | 有固定流程被 AI 承接；每个流程有负责人；费用仍自动归属                    | 为低频场景做复杂应用；绕过统一自动计费 |
